This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build

Requires `clang++` and GNU `make`. There is no code-generation build step; the precomputed bitboard lookup tables in `src/generated_data.cpp` are `const` globals computed once at program startup.

```bash
make          # full build â†’ build/pawnstar
make check    # build and run all test suites
make tools    # build helper tools â†’ build/stamp_net, build/filter_book
make doc      # generate Doxygen HTML into doc/html
make clean    # remove build artifacts and generated docs
```

Compiler flags include `-mbmi2` and `-mavx2` (BMI2 `pext` for slider attacks and AVX2 for the NNUE kernels; both present on Intel Haswell / AMD Zen 1+), `-std=c++20`, and `-I src`. Debug builds (`DEBUG=1`) add `-fsanitize=undefined,address`. ThinLTO is on by default (`LTO=0` to disable, forced off for `DEBUG`). Warnings are on (`-Wall -Wextra -Wpedantic`) but not fatal by default; `make WERROR=1` adds `-Werror` (for CI).

Always run `make clean && make` when switching branches or after changes to generated files â€” stale `.d` dependency files can cause spurious build failures.

## Running Tests

Tests are standalone executables in `test/`, linked against the engine objects but with their own `main()`. Build and run all tests with:

```bash
make check    # builds tests + tools, then runs all six suites; returns non-zero on any failure
```

Individual test executables:

```bash
./build/test_see              # 24 static exchange evaluation cases
./build/test_perft            # move generation regression (standard PERFT positions)
./build/test_bratko_kopec_nnue nnue/pawnstar-v9.bin   # 24 Bratko-Kopec positions searched with NNUE; reports moves solved
./build/test_nnue nnue/pawnstar-v9.bin test/nnue_reference.txt   # NNUE inference cross-check vs trainer
./build/test_nnue_incremental nnue/pawnstar-v9.bin              # incremental accumulator == full refresh
./build/test_opening_book     # opening-book parse/lookup regression
```

The six suites are `test_perft`, `test_see`, `test_bratko_kopec_nnue`, `test_nnue`, `test_nnue_incremental` and `test_opening_book`. `make check` depends on `tests tools`, so it also builds the helper tools (`stamp_net`, `filter_book`) â€” a broken tool fails the check rather than rotting unnoticed.

`make check` runs the NNUE tests against the shipped `nnue/pawnstar-v9.bin` (the Makefile passes the net via `$(wildcard â€¦)`, so the checks degrade to a green no-op only if the net/reference are absent):

With no arguments it is a no-op.
- `test_nnue_incremental` (`test/nnue_incremental.cpp`) takes `<net.bin>` and asserts the incrementally-maintained accumulator evaluates bit-identically to a full refresh over random move sequences (no-op without a net).

The 24 positions and their reference best moves live in the `kCases` array in `test/bratko_kopec_nnue_test.cpp`. The former score-based `test_bratko_kopec` and the `test_pawn_structure` suite tested the removed handcrafted evaluation and were deleted with it.

## Architecture

Pawnstar is a UCI chess engine using bitboard board representation and parallel alpha-beta search.

### Board representation

`Position` ([position.h](src/position.h)) is the central state class. It stores:
- A public `pieces[7]` array of per-piece-type bitboards (indexed by `Piece`; index 0 / `kNone` holds the occupied-squares set) and a public `colors[2]` array of per-color bitboards.
- A `squares_[64]` array for O(1) squareâ†’piece lookup.
- Zobrist hash, castling rights (`CastlingRights`), en passant square, half-move clock, and a checkers bitboard (squares giving check to the side to move).
- A public `bool is_null_move` (null-move marker) and a public `Color color_to_move` (side to move), the latter declared last so it occupies what was otherwise a trailing padding byte (`sizeof(Position) == 160`).

`Position` is immutable after construction. `MakeMove` and `MakeNullMove` return a new `Position` by value. `GenerateLegalMoves` / `GenerateLegalCaptures` dispatch to the templated `GenMoves<Color, bool>()`, which uses `Pins` to filter pseudo-legal moves. `pieces`/`colors`/`is_null_move`/`color_to_move` are public data members read directly (no getters); UCI does not declare game results, so there are no `IsGameOver`/`IsCheckmate`/`IsStalemate` helpers.

`Bitboard` ([bitboard.h](src/bitboard.h)) wraps a `uint64_t` with bitwise operations and a range-based iterator over set bits (`Square` values).

### Move encoding

`Move` ([move.h](src/move.h)) packs all move information into 64 bits:
- Bits 0â€“5: to square; 6â€“11: from square; 12â€“14: moving piece; 15â€“17: captured piece; 18â€“21: move type; 22: is-checking flag.
- Bits 23â€“24: TT node type; 25â€“32: TT depth (int8_t); 33â€“40: TT age (uint8_t); 41â€“63: score (signed 23-bit).

The score field (bits 41â€“63) is shared: during move ordering it holds the sort score, and in a stored TT entry it holds the search score. The two never coexist â€” a move being ordered in a move list is never simultaneously a TT entry. The TT metadata bits (23â€“63) let a transposition entry be just `{hash, move}` (128 bits); see the transposition table section. `Move::WithTTData`/`SetTTAge` pack the metadata and `Bits`/`FromBits` round-trip the raw 64-bit value for the lockless TT.

Move types: `kNonCapture`, `kCapture`, `kPawnDoublePush`, `kEpCapture`, `kCastling`, and promotion variants `kPromotionKnight/Bishop/Rook/Queen`.

`MoveList` is a `StackList`: a fixed-capacity, stack-allocated container, so move generation does no heap allocation. (The search's principal-variation lists (`Variation`) and the per-thread position stack are `std::vector`s â€” the position stack is reserved up front in the `SearchState` constructor; see Per-thread state.)

### Attack generation

`Attacks` ([attacks.h](src/attacks.h)) uses BMI2 `_pext_u64` for sliding-piece attacks: for each square the occupancy bits relevant to that slider are extracted with `pext`, used to index a per-square attack table. Tables live in `src/generated_data.cpp` as `const` globals computed once at program startup (no build-time codegen). The non-PEXT ray-scan fallback was removed â€” the build always targets BMI2 (`static_assert(__BMI2__)`).

Knight, king, and pawn attack tables are plain 64-entry arrays in `generated_data.cpp`.

`Pins` ([pins.h](src/pins.h)) computes absolute pins and allowed move masks before move generation by scanning enemy sliding pieces that see the king through exactly one friendly piece.

### Search

The search functions were consolidated onto the classes that own their state: `Game::SearchRootNode` lives in [game.cpp](src/game.cpp), and `Search`/`SearchSingleMove`/`SearchQuiescent`/`IterativeDeepen` are `SearchState` members in [search_state.cpp](src/search_state.cpp). The old standalone `search.h`, `search_alphabeta.cpp`, `search_quiescent.cpp` and `search_root_node.cpp` no longer exist.

**Iterative deepening** â€” `Game::SearchRootNode` ([game.cpp](src/game.cpp)) drives iterative deepening from `kStartDepth` (3) upwards, maintaining a principal variation. The per-iteration loop is `SearchState::IterativeDeepen`, which every Lazy SMP thread runs. It runs on a dedicated worker thread started by `Game::StartThinking`; the UCI `stop` command sets `game.is_cancel_pending`.

**Alpha-beta** â€” `SearchState::Search` ([search_state.cpp](src/search_state.cpp)) is a fail-soft negamax with:
- Transposition table probe/store (exact PV, lower-bound CUT, upper-bound ALL).
- Principal variation search (PVS): full window on the first move, null-window scout on the rest, re-search on a fail-high.
- Reverse futility pruning (a.k.a. static null move): at a non-PV node that is not in check, with `depth <= kRfpMaxDepth` (7) and outside the mate zone, if `static_eval - kRfpMargin (80) * depth >= beta` the node is cut and `static_eval` returned. The static eval is computed **once** per node (in `Search`) and reused by both RFP and null-move pruning. SPRT-tested **+114.8 Â± 22.5 Elo at 8+0.08** â€” the single biggest search win to date (pawnstar previously lacked the static-eval pruning suite). Costs a little shallow-depth tactical sharpness (Bratko-Kopec 20/24 at depth 8, recovering to 24/24 by deeper search) but the depth it buys at fixed time wins decisively.
- Null-move pruning.
- Late-move reduction (LMR): reduces any late move (after the 4th) outside a check sequence at depth > 2, with an extra reduction after the 7th (depth > 3), and a third reduction if that very-late move also has zero history count (SPRT-tested: ~neutral at fast 8+0.08, +11 Elo at 40 moves/min â€” a time-control-dependent gain). (Each thread searches its tree sequentially â€” there is no in-tree split.)
- Late-move (move-count) pruning (LMP): at a non-PV node not in check with `depth <= kLmpMaxDepth` (8), once `move_index >= kLmpBase (3) + depth*depth`, the remaining **quiet, non-checking** moves are skipped entirely (captures/promotions/checks are never pruned). Where LMR *reduces* late quiets, LMP *drops* them â€” a big effective-branching-factor cut at the frontier. SPRT-tested **+18.6 Â± 9.0 Elo at 8+0.08** on top of RFP. Like RFP it costs a little shallow-depth tactical sharpness that deeper search recovers (the Bratko-Kopec accepted sets were regenerated for it).
- Killer move heuristic (2 killers per ply): a quiet move that causes a beta cutoff is stored via `SearchState::RecordKiller`.
- History heuristic (`HistoryTable`) for move ordering â€” **per-thread** (owned by `SearchState`), so there is no cross-thread contention on its counters.
- Countermove heuristic + 1-ply continuation history (both **per-thread**, owned by `SearchState`, keyed by the previous move's (piece, to-square)): the countermove is the best quiet refutation to the previous move; the continuation history grades how often a quiet move was good as a follow-up to the previous move. The previous move is threaded into `Search`/`ScoreAndSortMoves` as a `prev_move` parameter (Move::None at the root and after a null move).

**Move ordering** â€” `SearchState::ScoreAndSortMoves` assigns each move a 23-bit ordering score stored in the `Move` itself, in descending bands: winning/equal captures and promotions (`kWinningCaptureBase` + SEE score), the two killer moves for the ply (just below winning captures), the countermove (just below the killers), quiet moves (main history count + 1-ply continuation history, clamped below the countermove), and finally losing captures (their negative SEE, sorting below quiet moves). The TT move is searched first, before the list is generated and sorted.

**Quiescence search** â€” `SearchState::SearchQuiescent` ([search_state.cpp](src/search_state.cpp)) searches captures only (no transposition table â€” it does not probe or store). It applies SEE pruning: because the move list is SEE-sorted by `ScoreAndSortMoves`, the loop reads each move's precomputed ordering score and **exits on the first losing capture** (`move.score() < 0`) â€” every remaining move is also a losing capture, so none are searched (no per-move SEE recompute). Removing the q-table and switching to this sorted early-exit was SPRT-tested **+38.65 Â± 13.78 Elo at 8+0.08** (1318 games, LLR-accepted [-5, 0]).

**Parallel search (Lazy SMP)** â€” `Game::SearchRootNode` spawns `hardware_concurrency()` threads (capped at `kMaxSearchThreads`, 256), each running a complete independent `SearchState::IterativeDeepen` search of the root. There is no tree splitting and no work queue â€” threads cooperate only through the shared lockless transposition table, where one thread's entries prefill another's probes. Each thread has its own `SearchState` (including its own history table), so the only shared mutable state is the transposition tables and `is_cancel_pending`. The thread count can be overridden with the `PAWNSTAR_THREADS` environment variable (e.g. `PAWNSTAR_THREADS=1` for a deterministic single-threaded search â€” used to regenerate BK reference data). To stop helpers recomputing the same tree in lockstep, the main thread (thread 0) searches every depth while each helper follows a Stockfish-style iteration-skip schedule (`kSkipSize`/`kSkipPhase` tables in `search_state.cpp`) that skips selected depths. Only the main thread prints `info`, applies time management and returns the authoritative move; because helpers race ahead through the table, the result (move and exact score) is non-deterministic between runs.

### Per-thread state

`SearchState` ([search_state.h](src/search_state.h)) owns:
- A reference to the shared `Game` (TT, clock, cancellation).
- `history` â€” the **per-thread** `HistoryTable` (no cross-thread sharing).
- `killers[kMaxPly][2]` â€” killer moves.
- `countermoves_` and `cont_hist_` â€” the **per-thread** countermove table and 1-ply continuation-history table (keyed by the previous move's (piece, to-square); `cont_hist_` is heap-allocated, ~0.4 MB).
- `node_count` â€” nodes searched by this thread.
- `positions_` (`std::vector<Position>`, reserved to `kMaxPly + 4` in the constructor) â€” per-thread copy-make position stack for the search tree. Reserved once up front so no `push_back` during search reallocates (which would invalidate the `CurrentPosition()` reference and break incremental NNUE updates).
- `accumulator_` (`nnue::Accumulator`) â€” the NNUE accumulator for the tip position, maintained incrementally across make/undo (only while `game.NnueActive()`, i.e. a net is loaded â€” false only for tools/tests that build a `Game` without one, e.g. perft/SEE).
- `hash_stack_` (`std::vector<HashEntry>`, reserved in the constructor to game length + `kMaxPly` + 4) â€” Zobrist hash history for 50-move/repetition detection. One reservation per search, so the per-node hot path still allocates nothing.

Lazy SMP retired the old YBW machinery: `SearchStatePool` and `ThreadPool` (and their `kSearchStatePoolCapacity`/`kThreadPoolQueueCapacity` constants) no longer exist. Threads are plain `std::thread`s spawned per search in `SearchRootNode`, each constructing its own `SearchState` on its stack.

### Transposition table

One table: 64 MB main (`kHashtableMegabytes`). (Quiescence search has no table of its own.) Each entry is exactly 128 bits: a 64-bit Zobrist `hash` and a 64-bit `move` whose spare bits encode the score, depth, node type (CUT/ALL/PV) and age. `Transposition` exposes `score()`/`depth()`/`node_type()`/`age()` accessors that read those packed bits.

The table is **lockless** (Hyatt's XOR trick). Each cell (`AtomicEntry`) stores two `std::atomic<uint64_t>`: `key = hash ^ data` and `data` (the packed move). A reader accepts a cell only when `key ^ data == probe hash`, so a torn pair written by concurrent stores is detected and read as a miss. Words are accessed with `relaxed` ordering; the XOR check supplies the cross-word consistency that relaxed ordering does not. Writers race benignly (last-writer-wins) â€” no spinlocks.

`TranspositionTable::Prefetch(hash)` issues a `__builtin_prefetch` for the cell a future probe will read; `SearchSingleMove` calls it right after `PlayMove` so the (random-access) line is resident by the time the recursive node probes the table. Result-neutral; pure latency hiding.

Aging is O(1): the table holds a `generation_` counter that `Age()` increments before each search; an entry is stale when its `age != generation_`, and `RecordTransposition` stamps the current generation when it writes. Replacement policy: prefer replacing stale entries, lower-depth entries, or any entry for PV nodes.

### Evaluation

`EvaluatePosition` ([evaluation.cpp](src/evaluation.cpp)) is **NNUE-only**: after the draw short-circuit (`kDrawScore` for material/fifty-move/repetition draws) it returns the network's score for the incrementally-maintained accumulator, memoised by Zobrist hash in the net-specific eval cache, then **scaled toward draw by the fifty-move clock** (`ScaleByRule50`: identity for `ReversibleMoveCount() <= 20` plies, then a linear ramp to 0 as the clock approaches 100). NNUE inputs are piece placements only, so without this a winning eval is identical at clock 0 and 99 and the search has no gradient to prefer progress (a clock-resetting pawn move/capture) over shuffling in a won-but-unconverted position. The scaling is applied **after** the eval cache (the cache stores the raw, clock-independent eval; the Zobrist hash excludes the clock). SPRT-measured **Elo-neutral at 8+0.08** (âˆ’0.25 Â± 3.08, 25k games) â€” kept as a non-regressing correctness fix (it stops the engine drawing won endgames by the fifty-move rule; the gain is in longer TC / vs weaker opponents, which equal-strength fast self-play can't see). It does **not** help cases with no clock-resetting move available (e.g. KBN-vs-K, where every move raises the clock so the scaling has no gradient â€” that needs mating technique/depth, a separate matter). NNUE is the engine's only evaluator â€” a loaded net is required to search; `main.cpp` loads the shipped net at startup, falls back to a copy embedded in the binary if the file can't be loaded, and exits only if both fail. The former handcrafted evaluation (material, piece-square tables, pawn structure, king safety, mobility, tapering) and its `EvaluateMaterial`/`DeterminePawnStructure`/etc. functions, the PST/mask tables in `evaluation.h`, and the HCE-only generated bitboards (`kKingAttacks2`, `kKingPawnShelter*`, the pawn-structure masks) were all removed when NNUE became the sole evaluator. `evaluation.h` now declares only `EvaluatePosition`.

SEE (static exchange evaluation, [static_exchange_evaluation.h](src/static_exchange_evaluation.h)) resolves a capture sequence on a square; `EvaluateStaticExchange` returns `{score, is_checking}`. It is independent of the NNUE eval: it orders captures and promotions in `ScoreAndSortMoves` (see Move ordering above), prunes losing captures in quiescence search, and is exercised by `test_see`.

### NNUE evaluation

`nnue::Network` ([nnue.cpp](src/nnue.cpp), [nnue.h](src/nnue.h)) is the engine's learned evaluation â€” the only evaluator. The `Game` owns one `nnue::Network` instance (accessors `NnueNetwork()`/`NnueActive()`); there are no nnue globals. `NnueActive()` is simply "a net is loaded" â€” true in normal operation (the engine loads the shipped net, or its embedded fallback, at startup and exits only if both fail), false only for tools/tests that construct a `Game` without a net (e.g. perft/SEE), where the accumulator is not maintained. After the draw short-circuit `EvaluatePosition` evaluates with the net.

- **Architecture**: a perspective net with bullet's `ChessBuckets` feature set â€” `768` inputs per (perspective, king bucket) â†’ `1024` feature transformer (shared across both perspectives), SCReLU activation, `concat[stm | ntm] = 2048 â†’ 1` output (centipawns, side-to-move relative). The shipped net (**v9**) uses **4 king buckets** selected by king file-pair (`kKingBucketMap`). Constants: `kHiddenSize=1024`, `kNumKingBuckets=4`, `kFeatureRows=768*4`, `kQA=255`, `kQB=64`, `kScale=400`.
- **int8 output layer (shipped)**: the engine's eval (`Network::Evaluate`) runs the output dot in **int8** â€” uint8 SCReLU activations (`clamp(x,0,QA)Â² >> kInt8Shift`, `kInt8Shift=9` â†’ `[0,127]`) Ã— int8 output weights, accumulated in int32. ~1.86Ã— faster than the exact int16 dot (a measured **+31.8 Â± 10.3 Elo at 8+0.08**), at a bounded ~26 cp deviation. It runs on the baseline AVX2 target via `pmaddubsw`; `S=9` keeps every pair within int16 so the AVX2 and VNNI (`vpdpbusd`, build `VNNI=1`) kernels are saturation-free and bit-identical. The exact int16 SCReLU forward pass remains as `Network::EvaluateExact` â€” the reference that `test_nnue` cross-checks against bullet (Â±2 cp) and that the int8 path approximates. (The feature transformer stays int16 â€” the int8 *feature*-weight experiment was measured and **removed** after a definitive âˆ’8 Elo SPRT; see [nnue/int8_quant_study.md](nnue/int8_quant_study.md).)
- **King buckets**: the shipped net (v9) uses **4 banks** selected by king **file-pair** (`kKingBucketMap` = `(file/2)`: files a/bâ†’0, c/dâ†’1, e/fâ†’2, g/hâ†’3 â€” file-based bucketing is preferable to quadrant/rank-based, since king safety and the castled-king pawn shelter are file-dependent and back-rank kings exercise all four banks). Each perspective selects its bank by *its own* king's square (in that perspective's orientation) via the 64-entry `kKingBucketMap`, kept byte-identical to the `ChessBuckets::new` array in `tools/bullet/pawnstar.rs`/`pawnstar_eval.rs`.
- **Feature index** (per perspective): `bucket*768 + colour*384 + (piece_type-1)*64 + square`, with the not-to-move perspective swapping the colour offset and flipping the square with `kRankFlip`. This matches bullet's `ChessBuckets::map_features`.
- **Accumulator** (`nnue::Accumulator`, two perspectives): maintained **incrementally** on `SearchState` across make/undo. `Network::Update`'s common case (no king-bucket crossing) finds the changed squares straight from the parent/child **colour bitboards** â€” a move touches only 2â€“4 squares, with captures/EP/castling/promotion all handled uniformly â€” and applies just those feature-column deltas to both perspectives (no per-piece-type scan). A king move that **crosses a bucket boundary** changes that whole perspective's bank, so `Update` rebuilds that one perspective from scratch (`RefreshSide`) and diffs the other (`DiffSide`) â€” at most one king moves per ply, so at most one side refreshes (with the shipped 4-bucket net, a king move across a file-pair boundary triggers this). `Network::Refresh` rebuilds both at the root (seeded with `feature_bias`). The colour-XOR delta is bit-identical to a per-piece diff (same columns; int16 accumulator add/sub is order-independent), and `test_nnue_incremental` checks the incremental accumulator stays bit-identical to a full refresh (random trees, including bucket crossings on multi-bank nets).
- **Weights** are bullet's native quantised payload (tightly packed little-endian `int16`: `feature_weights[768*1*1024]` bucket-major, `feature_bias[1024]`, `output_weights[2048]`, scalar `output_bias`; bullet pads the trailing tensor to 64 bytes). `Load` also requantises the output weights to int8 (`output_weights_i8_`, scale `output_w_scale_` = 1 for v8 since `|w|â‰¤127`) for the shipped int8 forward pass. The exact int16 forward pass and dequantisation mirror bullet's example; correctness is verified by `test_nnue` (`EvaluateExact` vs bullet, int8 `Evaluate` vs `EvaluateExact`).
- **Self-describing header**: every loadable net is prepended with a 32-byte `NetHeader` (magic `PSN1` + the arch params `kInputSize`/`kNumKingBuckets`/`kHiddenSize`/`kQA`/`kQB`/`kScale`; see `src/nnue.h`). `Network::Load` **requires the header**: it rejects any net without the `PSN1` magic (a raw/unstamped bullet net) and validates a stamped file's arch fields against the engine's constants, **rejecting a mismatch** (so an incompatible net can't be silently misread). The old headerless-raw acceptance path was removed â€” a raw bullet net must be stamped first. Stamp a raw net with `tools/stamp_net` (`make tools`) â€” it embeds the building engine's constants, so it only stamps a matching net (and only accepts a raw, not already-stamped, net).
- **Eval cache**: an 8 MB lockless `EvalCache` (owned by the `Game`) memoises the net-specific static eval by Zobrist hash (single 64-bit atomic per entry: 48-bit key + 16-bit score), so a cache hit in `EvaluatePosition` skips the forward pass. Cleared on `NewGame` and on `EvalFile` load (evals are net-specific). ~42% hit rate / ~5% faster.
- **Loading**: NNUE is the only evaluator, so a net is **required** â€” `main.cpp` loads `nnue/pawnstar-v9.bin` (cwd-relative, like the book) at startup. If the file can't be loaded (absent, wrong cwd, wrong architecture) it **falls back to a copy of the net embedded in the binary** (`src/embedded_net.S` `.incbin`'s the net at the Makefile's `EMBED_NET` path, passed in via `-DEMBED_NET_PATH` so the embedded copy can't drift from the shipped net; linked into the engine only), so the engine always has a working evaluator; it exits only if **both** the file and the embedded fallback fail (a corrupt build). Override the path via env (`PAWNSTAR_EVALFILE=<path>`) or load a different net at runtime via UCI `setoption name EvalFile value <path>`. The embedded fallback is loaded via `Network::LoadFromMemory`, which shares the exact header detection/validation of `Network::Load`. There is no `UseNNUE` option (nothing to toggle to). The network is a single `nnue::Network` owned by the `Game`, read-only after load and shared by all Lazy SMP threads, which call only its `const` methods.

**Training** (see [nnue/README.md](nnue/README.md) and [tools/](tools)): the shipped nets are trained on **public PlentyChess** bulletformat data (strong-engine labels â€” label quality, not self-play volume, was the lever) via the bullet trainer (`tools/bullet/pawnstar.rs`, pinned commit installed by `tools/setup_bullet.sh`). (Self-play data generation â€” `gen_data.cpp` / `run_gendata.sh` / `make_openings.sh` â€” was the original bootstrap source and has been removed.) Minimal flow: `tools/setup_bullet.sh` â†’ `OPENINGS=<openings.epd> tools/run_experiment.sh <shuffled.data> <net.bin>` (trains via `tools/train_pipeline.sh`, verifies via `tools/verify_net.sh`, then SPRTs via `tools/run_sprt.sh`). GPU training needs a CUDA toolkit and a driver supporting CUDA â‰¥ 12.3 (bullet uses `cuFuncLoad`); set `BULLET_FEATURES=""` to train on CPU. NNUE strength is measured by game play (SPRT, now always net-vs-net since the HCE baseline is gone), not by the Bratko-Kopec suite.

### Opening book

`OpeningBook` ([opening_book.h](src/opening_book.h)) is loaded from `pawnstar.book` in the working directory at startup (`main.cpp`); the shipped book ([pawnstar.book](pawnstar.book)) is the public-domain [TSCP](https://github.com/terredeciels/TSCP) `book.txt`. The format is plain text, one opening line per row, each move in **coordinate notation** (e.g. `e2e4 e7e5 g1f3`); `ParseLineOfPlay` replays each line from the start position, ignoring tokens that begin with a digit (move numbers) and treating `#` as a to-end-of-line comment. The book is a `std::map<zobrist_t, std::vector<Move>>` storing moves *with repeats*, so `GetMove`'s uniform-random pick is frequency-weighted. `SearchRootNode` returns a book move immediately on a hit, before searching; the `bookmoves` UCI command lists moves and frequencies for the current position. There is no SAN/PGN parser â€” coordinate notation only.

### UCI protocol

[input_handling.cpp](src/input_handling.cpp) parses all UCI commands plus engine-specific diagnostics: `eval` (print the NNUE position evaluation), `nnue` (print the raw NNUE eval), `getboard` (print board), `bookmoves` (list opening-book moves for the current position), `dbg` (dump debug counters). `setoption` handles `EvalFile` (string, path to a net), advertised in the `uci` response. (There is no `UseNNUE` option â€” NNUE is the only evaluator.)

### Generated data

[generated_data.cpp](src/generated_data.cpp) holds the precomputed lookup tables (attack/ray bitboards, pext sliding-attack tables, intervening-squares masks, Zobrist hashes) declared `extern const` in [generated_data.h](src/generated_data.h). They are computed once at program startup (dynamic initialisation of the `const` globals) by the pure helper functions at the top of that file â€” there is no separate generator program or build step. Edit `generated_data.cpp` when the table logic changes. (Historically these were emitted as literals by a standalone `generate_constants.cpp`, since removed.)
